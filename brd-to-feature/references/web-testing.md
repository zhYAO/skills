# Web 前端测试（Chrome MCP）

用于阶段 6 测试 Web 项目。Chrome MCP 工具名形如 `mcp__Claude_in_Chrome__*`（不同环境前缀可能不同，按实际可用工具名为准）。

## 缺失检查 + 连接

进入 Web 测试前，先确认 Chrome MCP 是否已连接（列出可用工具，找 `mcp__Claude_in_Chrome__*` 之类）。

和 markitdown / 小程序 MCP 不同，**Chrome MCP 走的是浏览器扩展 + 授权，无法由 agent 命令行自动安装**——这步必须用户手动做。若没连接，明确引导用户：

1. 在 Chrome 应用商店安装 "Claude in Chrome"（或所用 agent 对应的浏览器扩展）。
2. 打开扩展并登录/授权，把它连接到当前 agent。
3. 完成后回来，agent 重新检测工具即可继续。

引导完成、确认工具可用后再进入下面的准备步骤。若用户暂时无法连接，可改用其它已连接的浏览器自动化方案，或把这部分用例标记为"需人工验证"。

## 准备

1. 按项目脚本启动 dev server（如 `npm run dev` / `pnpm dev`），拿到本地 URL（如 http://localhost:5173）。
2. 确认 Chrome MCP 已连接（见上）。

## 常用能力

- 导航到页面（navigate）。
- 读取页面内容（read_page / get_page_text）——用于断言文案、元素是否存在。
- 操作页面（computer / form_input）——点击、输入、提交表单。
- 读取控制台报错（read_console_messages）——抓 JS 异常、警告。
- 读取网络请求（read_network_requests）——核对接口是否正确发出、状态码、返回。

## 执行每条用例

按阶段 5 用例的"操作步骤"逐步操作，对照"预期结果"断言：

1. 导航到目标页面。
2. 执行操作（点击/输入/滚动/提交）。
3. 读取页面文本与状态，核对功能预期。
4. UI 还原类用例：对照 Figma 设计稿核对布局、间距、配色、字号等关键视觉。
5. 读 console 与 network，确认无报错、接口正常。

## 发现问题 → 自愈

- console 报错 / 接口失败 / 渲染不符 / 逻辑错误 → 定位到代码 → 修复 → 重跑该用例。
- 记录每轮"问题 / 根因 / 修复 / 复测结果"，供回归文档使用。
- 同一用例尝试上限（建议 3 次）仍不过，停下来报告用户，不强行标记通过。

## Chrome MCP 连接失败处理

按严重程度分级处理（与 SKILL.md 红名单 #11 配合）：

1. **工具列表里找不到 `mcp__Claude_in_Chrome__*`**：浏览器扩展未安装或未授权，提示用户装扩展 + 授权 + 重连（具体步骤见上"缺失检查 + 连接"）。
2. **工具在但导航/操作失败**：可能是扩展已断连或页面状态异常，先 `read_console_messages` 看是否有扩展侧报错，再请用户重新授权。
3. **连续 3 次操作失败**：🔴 **STOP** 上报用户，禁止自行跳过 Chrome MCP 改用静态分析并继续产出报告。
4. **降级选项**：若用户明确同意降级为静态分析，**必须**在产出物顶部显式标注"运行通道：⚠️ 仅静态分析"，并在遗留事项里列出运行时复测脚本（Chrome MCP 恢复后补测）。
